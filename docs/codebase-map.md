## 7. Debugging Zones & Risk Flags

### 🔥 Hotspot Files (Most Frequently Modified)

| File | Modification Frequency | Risk Level | Notes |
|---|---|---|---|
| `components/shared/heros/HeroSection.tsx` | Very High | 🔴 Critical | GSAP + Lenis + scroll transition logic |
| `app/page.tsx` | High | 🔴 Critical | Section ordering + scroll orchestration |

| `components/shared/navbar/` | Medium | 🟠 High | Mobile menu + scroll state |
| `components/shared/eco-events-form.tsx` | Medium | 🟡 Medium | 7KB — complex form schema |
| `components/shared/eco-producer-form.tsx` | Medium | 🟡 Medium | 8KB — largest single component |
| `tailwind.config.ts` | Low-Medium | 🟡 Medium | Theme changes cascade everywhere |
| `app/globals.css` | Low | 🟡 Medium | CSS variable changes cascade |

### ⚠️ Known Architectural Risks

1. **No API routes** (`/app/api/` missing) — Forms have no server-side handler. Data submitted via forms either goes to an external service or is not captured. **Action required: add API routes or confirm external service.**

2. **Animation orchestration complexity** — Three animation libraries (GSAP, Lenis, Framer Motion) coexist. Conflicts can arise when ScrollTrigger and Lenis dispute scroll ownership. **Always refresh ScrollTrigger after Lenis updates.**

3. **HeroSection is a single point of failure** — It initialises Lenis scroll for the entire page via `onTransitionComplete`. If this callback is broken, all subsequent sections lose smooth scroll.

4. **No error boundaries** — Zero `<ErrorBoundary>` components detected. A runtime error in any client component can crash the full page.

5. **No loading.tsx / error.tsx** — Missing Next.js App Router conventions for loading and error states on route segments.

6. **Typo in shared component** — `eligble-popup.tsx` should be `eligible-popup.tsx`. Low priority but indicates need for naming convention enforcement.

7. **No state management layer** — For a growing feature set (market, community, events), consider Zustand or React Context if cross-component state grows.

### 🛡️ Risky Change Zones

| Zone | Risk | Why |
|---|---|---|
| `app/layout.tsx` → metadata changes | High | Affects SEO across all pages simultaneously |
| `tailwind.config.ts` → theme token changes | High | Cascades to every component without TypeScript safety |
| `HeroSection.tsx` → scroll trigger config | High | Can break Lenis/GSAP integration silently |
| `globals.css` → CSS custom property rename | High | No compile-time check catches broken `var()` references |
| `lib/utils.ts` → `cn()` signature change | High | Called in every component |

## 8. Architecture Summary

### Pattern: Feature-Sliced Shared Components

EcoCan follows a **page-driven composition** model where:
- Each route in `app/` is a thin orchestrator
- Business logic lives in `components/shared/` feature components  
- Primitive UI lives in `components/ui/` (Radix + Shadcn wrappers)
- Utilities live in `lib/`

This is a **monolithic frontend** with no micro-frontend or monorepo sub-package split (yet — `pnpm-workspace.yaml` is present as a stub).

### Scroll Architecture

```
Lenis (physics layer)
    └── GSAP ScrollTrigger (progress consumer)
            ├── Hero exit animation (scrub: 0.6)
            └── Section reveal animations (.ps-animate)
```

Lenis is initialised inside `HeroSection` after `onTransitionComplete` fires from `app/page.tsx`. This is a **tight coupling** — the scroll system depends on the hero completing its render cycle.

### Styling Architecture

```
Tailwind CSS (utility classes)
    + tailwind-merge (conflict resolution)
    + clsx (conditional classes)
    + class-variance-authority (component variants)
    = cn() helper in lib/utils.ts
```

CSS custom properties in `globals.css` provide theming tokens. `tailwind.config.ts` extends the default palette with brand colours.

## 10. Auto-Update System Design

### 10.1 Change Detection Strategy

#### Git Hooks (Local)

| Hook | Trigger | Action |
|---|---|---|
| `post-commit` | After every local commit | Runs `node scripts/generate-codebase-map.mjs`, stages updated docs |

#### CI/CD (Remote)

| Trigger | Scope | Action |
|---|---|---|
| `push` to any branch | All file changes | Runs full or incremental map generation |
| `pull_request` | Changed files only | Incremental update + diff comment |

#### File Watchers (Local Dev)

For real-time local updates, add to `package.json`:
```json
"map:watch": "node --watch scripts/generate-codebase-map.mjs"
```

### 10.2 Significant Change Definition

| Change Type | Qualifies? | Detection Method |
|---|---|---|
| New file added | ✅ Yes | `git diff --name-status` shows `A` |
| File deleted | ✅ Yes | `git diff --name-status` shows `D` |
| Component renamed/moved | ✅ Yes | `R` status in git diff |
| Dependency added/removed | ✅ Yes | `package.json` SHA changed |
| New page route | ✅ Yes | New file in `app/**/page.tsx` |
| API route added | ✅ Yes | New file in `app/api/**/route.ts` |
| Shared component modified | ✅ Yes | SHA change in `components/shared/` |
| Tailwind config changed | ✅ Yes | `tailwind.config.ts` SHA changed |
| CSS var rename | ✅ Yes | `globals.css` SHA changed |
| Test file changed | ❌ No | `__tests__/` excluded from map |
| Lock file changed | ❌ No | `pnpm-lock.yaml` excluded |
| README/docs changed | ❌ No | `.md` files (except codebase-map) excluded |

### 10.3 Incremental Update Logic

```
1. Read previous map hash from docs/codebase-map.json
2. Run: git diff --name-status HEAD~1 HEAD
3. Filter changed files against significance rules above
4. If no significant changes → skip regeneration (exit 0)
5. If significant changes:
   a. Re-scan ONLY affected directories
   b. Update relevant sections of the map
   c. Append changelog entry with commit hash + timestamp
   d. Write updated docs/codebase-map.md + docs/codebase-map.json
   e. Stage and commit (CI) or stage only (local hook)
```

### 10.5 Performance Optimization

- **File-level SHA hashing**: Each file's SHA is stored in `codebase-map.json`. On next run, only files with changed SHAs are re-analysed.
- **Directory-level diffing**: Entire sub-trees with no SHA changes are skipped.
- **Dependency graph diffing**: `package.json` SHA is compared; if unchanged, the dependency section is copied verbatim.
- **Max cost**: Full rescan of `components/` + `app/` takes <2 seconds on this repo size.

## 12. CI/CD Workflow

Full workflow at `.github/workflows/codebase-map.yml`:

```yaml
name: Update Codebase Map

on:
  push:
    branches: ['**']
    paths-ignore:
      - 'docs/codebase-map.md'
      - 'docs/codebase-map.json'
      - '**.md'
      - 'pnpm-lock.yaml'
      - 'package-lock.json'
  pull_request:
    branches: [main]

jobs:
  update-map:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 2
          token: ${{ secrets.GITHUB_TOKEN }}

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Generate codebase map
        run: node scripts/generate-codebase-map.mjs

      - name: Commit updated map
        if: github.event_name == 'push'
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add docs/codebase-map.md docs/codebase-map.json
          git diff --staged --quiet || git commit -m "docs(auto): update codebase map [skip ci]"
          git push
```

> **`[skip ci]`** in the commit message prevents an infinite CI loop.
